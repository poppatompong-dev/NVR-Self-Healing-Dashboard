📝 เส้นทางการพัฒนา (Project Timeline)

ขั้นตอนที่ 1: ตรวจพบปัญหาเวลาไม่ตรงกัน
ระบบทำงานแบบปิด (Isolated Network) ไม่มีอินเทอร์เน็ต ทำให้เวลาบนเซิร์ฟเวอร์ NVR และกล้องเกิดการคลาดเคลื่อนสะสม ส่งผลให้บันทึกภาพไม่ตรงกับความเป็นจริงและใช้เป็นหลักฐานยาก
ขั้นตอนที่ 2: วางระบบ NTP Server ลำดับขั้น
ตั้งค่า NVR 1 เป็นแม่ข่ายหลัก, NVR 2 ดึงเวลาจาก NVR 1 และกล้องดึงเวลาจาก NVR ของตัวมันเอง เพื่อให้เวลาเป็นระบบเดียวกันทั้งหมด
ขั้นตอนที่ 3: ปัญหาการอนุญาตสิทธิ์กล้อง NVR 2
กล้องกลุ่มที่ 2 ของ NVR 2 ปฏิเสธสิทธิ์การซิงค์เวลา ทำการทดสอบอย่างละเอียดจนพบรหัสลับที่ถูกต้องคือ username 'Admin' และ password '1234'
ขั้นตอนที่ 4: พัฒนาระบบรายงานรวมศูนย์ 10x Performance
เปลี่ยนระบบเช็กสถานะเป็นแบบขนานขจัดข้อจำกัดเรื่อง Network Jitter และรวมรายงานผลของกล้อง 33 ตัวเข้าอยู่ในอีเมลฉบับเดียว ส่งตรงประหยัดทรัพยากร

🔧 ปัญหาที่พบจริง แนวทางการแก้ไขเชิงระบบ (Issues & Solutions)

1. ปัญหาการปฏิเสธสิทธิ์เขียนกล้อง NVR 2 (HTTP 401 Unauthorized)

อาการ: ระบบสามารถดึงเวลาจากกล้องมาแสดงผลได้ปกติ แต่ไม่สามารถสั่งแก้เวลาในกล้องบางตัวได้ (พบ HTTP 401 ในกล้องกลุ่มใหญ่ของ NVR 2)
วิเคราะห์: เนื่องจากกล้องกลุ่ม B (16 กล้อง) ใช้เฟิร์มแวร์แบบ OEM ที่มีมาตรฐานความปลอดภัยแบบ Digest Authentication และมีรหัสคนละชุดกับกล้องหลัก
แนวทางแก้ไข: ได้ทำการทดสอบเจาะสิทธิ์ระบบเชิงรุกจนค้นพบรหัสที่สมบูรณ์ คือ username 'Admin' และ password '1234' และจัดทําการแมปสิทธิ์อัตโนมัติในโค้ด ปัจจุบันสามารถแก้ไขเวลาได้สมบูรณ์แล้ว

2. ปัญหากล้องออฟไลน์ชั่วคราวระหว่างสแกนขนาน (Network Jitter & Timeout)

อาการ: ในบางครั้งการเช็กสถานะพบกล้องหลุดการเชื่อมต่อ (Offline) แบบสุ่ม ทั้งๆ ที่เปิดเช็กเดี่ยวๆ ทำงานได้ปกติ
วิเคราะห์: ระบบตัวใหม่ทำงานเร็วมาก (Threaded Parallel) ทำการสแกนกล้อง 33 ตัวพร้อมๆ กันในเวลาเสี้ยววินาที สวิตช์หรือการตอบสนองของกล้องบางตัวเกิดความล่าช้าชั่วขณะจนเกินค่า Timeout เดิมที่ 1.5 วินาที
แนวทางแก้ไข: ขยายค่าหน่วงเวลาการตอบสนองของพอร์ตเชื่อมต่อ (Connection Handshake Timeout) ขึ้นเป็น 3.0 วินาที เพื่อความเสถียร ผลลัพธ์คือเช็กออนไลน์ได้แม่นยำ 100%

3. ปัญหาเวลากระโดดสูงเกินจนบริการ Web Server กล้องค้างชั่วขณะ (Time-Jump Delay)

อาการ: เมื่อเวลาของกล้องคลาดเคลื่อนหนัก (เช่น ย้อนอดีตไปหลายปี หรือต่างกันเป็นชั่วโมง) หลังซิงค์ระบบมักค้างชั่วขณะทำให้ Timeout การสั่งยืนยันค่าล้มเหลว
วิเคราะห์: เป็นธรรมชาติของเฟิร์มแวร์กล้อง เมื่อเวลาเปลี่ยนแปลงกะทันหันอย่างมีนัยสำคัญ ระบบหลังบ้านจะทำการรีเซต Session tokens และระบบบันทึกภาพใหม่ทันที ทำให้ตัวดึงข้อมูลค้าง
แนวทางแก้ไข: ขยาย Timeout สั่งซิงค์เวลาของกล้องเป็น 15 วินาที พร้อมเพิ่มการหน่วงเวลา 1.0 วินาทีให้กล้องบันทึกค่าและเคลียร์เซสชันก่อนเปลี่ยนโหมด ปัจจุบันสามารถรับส่งคำสั่งได้เสถียร

🌱 คู่มือความยั่งยืนของระบบในระยะยาว (Sustainability & Maintenance Guide)

ระบบการซิงค์เวลาและการรายงานผลนี้ ถูกออกแบบโครงสร้างให้มีความยั่งยืน สูญเสียทรัพยากรระบบน้อยที่สุด และเป็นระเบียบ เพื่อให้เจ้าหน้าที่ทั่วไปที่ไม่มีความรู้ทางเทคนิคเชิงลึก สามารถตรวจสอบ แก้ไข และดูแลระบบต่อไปได้อย่างมั่นคง ดังนี้ครับ:

1. แนวทางการบำรุงรักษาปกติ (Routine Check)

• ตรวจสอบรายงานวันละ 2 ครั้ง: ในเวลา 08:00 น. และ 17:00 น. ระบบจะส่งรายงานสถานะเวลารวมฉบับย่อทางอีเมล โดยให้อ่านที่กราฟ ASCII Charts หากมีแท่งสีแดง (Offline) หรือสีส้ม (Skewed) เกิดขึ้น ให้ส่งตารางด้านล่างให้ช่างเข้าตรวจสอบแก้ไขหน้างาน
• ระบบแจ้งเตือนฉุกเฉินทันที (Instant Alerts): หากมีกล้องตัวใดดับลงจากเครือข่าย ระบบแจ้งเตือนจะส่งสัญญาณเตือน (Alert) ทางอีเมลทันทีโดยไม่ต้องรอรอบรายงาน เพื่อการแก้ไขอย่างรวดเร็ว

2. การรับมือเมื่อเกิดไฟฟ้าดับหรือระบบขัดข้อง (Power Outage Recovery)

• ระบบสตาร์ทตัวเองอัตโนมัติ: ภารกิจควบคุมเวลาทั้งหมดจดทะเบียนไว้ในระดับระบบ Windows Task Scheduler ภายใต้บัญชีระบบพิเศษ `NT AUTHORITY\SYSTEM` ทำให้เมื่อเครื่อง NVR เปิดทำงานใหม่หลังจากไฟฟ้าดับ ภารกิจทั้งหมดจะเริ่มรันตัวเองเบื้องหลังอัตโนมัติทันทีโดยไม่ต้องล็อกอินเข้าระบบค้างไว้
• ความทนทานต่ออุปสรรคเครือข่าย: กรณีที่การสื่อสารระหว่าง NVR ขัดข้อง ตัว NVR 2 จะยังคงทำหน้าที่ซิงค์เวลากล้องของตัวเองอย่างมั่นคงจากสคริปต์โลคอลเบื้องหลัง และเมื่อระบบเน็ตเวิร์กกลับมาเชื่อมโยงปกติ ระบบรวมศูนย์ NVR 1 จะประสานงานเวลากลับมาโดยอัตโนมัติ

3. แนวทางปฏิบัติเมื่อต้องการแก้ไข/ปรับเปลี่ยนอุปกรณ์กล้อง (Adding New Hardware)

• กรณีกล้องชำรุดและเปลี่ยนตัวใหม่ (IP เดิม): หากกล้องเปลี่ยนทดแทนโดยใช้ไอพีเดิม ให้ตรวจสอบให้แน่ใจว่ารหัสผ่านและไอพีอยู่ในกลุ่ม credentials เดียวกันในระบบ หากเวลาคลาดเคลื่อนสคริปต์ระบบเงียบ 15 นาทีจะทำการดันเวลาให้ตรงทันทีโดยอัตโนมัติภายในรอบการทำงานถัดไป
• กรณีต้องการเพิ่มกล้องใหม่ในระบบ (New IP Address):
   1. เข้าไปแก้ไขสคริปต์รายงานหลัก `D:\avigilon_ntp_tools\unified_nvr_report.py`
   2. นำไอพีกล้องตัวใหม่ไปเติมลงในตัวแปรกล้องที่เหมาะสม (`NVR1_IPS` หรือ `NVR2_IPS`) และกำหนดรหัสผ่านในสคริปต์ `get_camera_creds(ip)` หากมีการเปลี่ยนชุดรหัสผ่านใหม่
